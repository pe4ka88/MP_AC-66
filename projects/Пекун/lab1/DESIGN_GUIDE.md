## 📐 ЦВЕТОВАЯ СХЕМА

### Выбор палитры Material Design 3

**Почему эта палитра?**

1. **Primary (#6750A4)** — элегантный фиолетовый
   - Современный и нейтральный
   - Не отвлекает от игрового процесса
   - Ассоциируется с мышлением и концентрацией

2. **Accent (#00B8D4)** — бирюзовый
   - Контрастирует с Primary
   - Привлекает внимание к важным элементам
   - Создает баланс теплых и холодных тонов

3. **Success (#4CAF50)** — зеленый
   - Интуитивно понятный цвет успеха
   - Используется для совпавших карточек
   - Положительная обратная связь

4. **Background (#FEF7FF)** — светло-фиолетовый
   - Мягкий, не напрягает глаза
   - Гармонирует с Primary
   - Создает ощущение уюта

## 🎴 ОФОРМЛЕНИЕ КАРТОЧЕК

### Анатомия карточки

```
┌─────────────────────┐
│  CardView Container │ ← Elevation: 8dp
│  ┌───────────────┐  │
│  │   ImageView   │  │ ← Padding: 16dp
│  │  (Изображение)│  │
│  └───────────────┘  │
└─────────────────────┘
     Radius: 16dp
```

### Состояния карточки

1. **Закрыта (Back)**
   - Градиент: #8B5CF6 → #6366F1
   - Border: 2dp, #E7E0EC
   - Elevation: 8dp
   - **Почему?** Градиент добавляет глубину, делает карточки живыми

2. **Открыта (Front)**
   - Фон: #FFFFFF
   - Border: 2dp, #E7E0EC
   - Изображение видно
   - **Почему?** Белый фон не конкурирует с содержимым

3. **Совпала (Matched)**
   - Градиент: #4CAF50 → #81C784
   - Border: 2dp, #4CAF50
   - **Почему?** Зеленый = успех, пользователь сразу видит прогресс

### Анимация Flip

```java
// Поворот на 90° (карточка исчезает)
imageView.animate()
    .rotationY(90f)
    .setDuration(150)
    
// Смена изображения

// Возврат с -90° до 0° (карточка появляется)
imageView.animate()
    .rotationY(0f)
    .setDuration(150)
```

**Почему 150ms?** Достаточно быстро, но не резко — глаз успевает уследить.

## 📱 UX-УЛУЧШЕНИЯ

### 1. Splash Screen (3 секунды)

**Зачем?**
- Время на загрузку ресурсов
- Демонстрация бренда/авторства
- Плавный вход в приложение

**Что показываем?**
- Название игры (крупно, жирно)
- Номер лабораторной работы
- ФИО и группу автора
- Прогресс-индикатор

**Анимация:**
- Fade In + Scale (0.8 → 1.0)
- Эффект "всплывания"
- 800ms длительность

### 2. Главное меню

**Почему карточки, а не кнопки?**
- Material Cards выглядят современно
- Больше места для текста и иконок
- Визуально группируют функционал
- Реагируют на нажатия (ripple effect)

**Структура:**
```
Заголовок
   ↓
[▶ Начать игру    ] ← Главное действие (сверху)
[📊 Статистика    ]
[👤 Об авторе     ]
   ↓
Подпись автора
```

### 3. Экран настроек

**Почему RadioButton?**
- Взаимоисключающие опции
- Пользователь видит все варианты сразу
- Стандартный Android-паттерн

**Группировка:**
- Логическое разделение (размер / режим)
- Заголовки секций жирным
- Достаточные отступы между группами

### 4. Игровой экран

**Информационная панель:**
```
┌────────────────────────────────┐
│       Игра "Память"            │
├────────┬─────────┬──────────────┤
│ Время  │Осталось │    Ходов     │
│ 00:42  │    5    │     12       │
└────────┴─────────┴──────────────┘
```

**Почему три метрики?**
- **Время** — основной показатель сложности
- **Осталось** — визуализация прогресса
- **Ходов** — дополнительная статистика

**Размещение кнопок:**
- Внизу экрана (zone of thumb)
- Две кнопки: Перезапуск + Выход
- Равная ширина (1:1)

### 5. Анимации и фидбек

**Что анимируем?**
- ✅ Переворот карточек (flip)
- ✅ Появление элементов (fade)
- ✅ Диалоги (scale)
- ✅ Ripple на нажатия

**Задержки:**
- Открытие карточки: мгновенно
- Проверка совпадения: 500ms
- Закрытие неправильных: 1000ms

**Почему 1000ms для ошибки?**
- Пользователь успевает запомнить карточки
- Не слишком долго (не раздражает)
- Не слишком быстро (успеваешь увидеть)

## 📏 ОТСТУПЫ И СЕТКА

### Базовая единица: 8dp

```
4dp  — Минимальный отступ (между элементами в группе)
8dp  — Стандартный отступ (карточки в grid)
12dp — Средний отступ (кнопки)
16dp — Большой отступ (padding контейнеров)
24dp — Отступы страниц (margin)
32dp — Разделение секций
```

### Скругления

- 16dp — Карточки, контейнеры
- 24dp — Кнопки (более плавно)

### Elevation (тени)

- 0dp — Фон
- 2dp — Плоские элементы
- 4dp — Header, Footer
- 8dp — Карточки в игре

## 🔄 ЛОГИКА И GAMEPLAY

### Почему 3 режима?

1. **Пары (2 карточки)**
   - Классика, всем понятна
   - Легкий уровень сложности
   - Быстрое прохождение

2. **Тройки (3 карточки)**
   - Усложнение: нужно запомнить больше
   - Увеличивает replay value
   - Тренирует память лучше

3. **Гибридный**
   - Непредсказуемость
   - Нельзя привыкнуть к паттерну
   - Максимальная концентрация

### Размеры поля

| Размер | Карточек | Пары | Сложность |
|--------|----------|------|-----------|
| 4×4    | 16       | 8    | ★☆☆       |
| 4×5    | 20       | 10   | ★★☆       |
| 6×6    | 36       | 18   | ★★★       |

## 💾 СТАТИСТИКА (SharedPreferences)

### Что сохраняем?

```java
{
  "total_games": 42,              // Всего игр
  "best_time": 87,                // Лучшее время (сек)
  "last_time": 132,               // Последнее время
  "pairs_games": 20,              // Игр в режиме пар
  "pairs_best": 87,               // Лучшее время (пары)
  "triples_games": 15,            // Игр в режиме троек
  "triples_best": 145,            // Лучшее время (тройки)
  "hybrid_games": 7,              // Игр в гибриде
  "hybrid_best": 112              // Лучшее время (гибрид)
}
```

### Почему SharedPreferences?

- ✅ Простота реализации
- ✅ Быстрый доступ
- ✅ Автоматическое сохранение
- ✅ Не требует дополнительных разрешений
- ✅ Подходит для небольших данных

**Альтернативы:** Room, SQLite (избыточно для этой задачи)

## 🎨 ВИЗУАЛЬНАЯ ИЕРАРХИЯ

### Принципы

1. **Контраст**
   - Важное — крупнее и ярче
   - Второстепенное — мельче и бледнее

2. **Группировка**
   - Связанные элементы рядом
   - Отступы показывают связи

3. **Выравнивание**
   - Все по сетке
   - Единый ритм

4. **Повторение**
   - Одинаковые элементы выглядят одинаково
   - Создает узнаваемость

## 🚀 ПРОИЗВОДИТЕЛЬНОСТЬ

### Оптимизации

1. **Анимации на GPU**
   - Используем property animations
   - Избегаем layout changes во время анимации

2. **Переиспользование Views**
   - GridLayout создается один раз
   - Не пересоздаем при рестарте

3. **Lazy Loading**
   - Drawable загружаются по требованию
   - Не храним бitmaps в памяти

4. **Handler вместо Timer**
   - Более эффективно для UI-потока
   - Автоматическая очистка
